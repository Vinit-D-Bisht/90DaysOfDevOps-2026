# Task 1: Understanding workflow_call

## 1. What is a Reusable Workflow?

A reusable workflow is a GitHub Actions workflow that can be called by other workflows. It allows teams to define common CI/CD logic once and reuse it across multiple repositories or workflows.

## 2. What is workflow_call?

`workflow_call` is a special trigger that allows a workflow to be invoked by another workflow.

## 3. Reusable Workflow vs Regular Action

Reusable Workflow:
1. Can contain multiple jobs.
2. Can define inputs, outputs, and secrets.
3. Runs as an entire workflow.

Regular Action:
1. Executes as a single step.
2. Usually performs one specific task.
3. Cannot contain multiple jobs.

## 4. Where Must a Reusable Workflow Live?

Reusable workflows must be stored inside:

```text
.github/workflows/
```

# Task 2: First Reusable Workflow

## File

```text
.github/workflows/reusable-build.yml
```

## YAML

```yaml
name: Reusable Build

on:
  workflow_call:
    inputs:
      app_name:
        required: true
        type: string

      environment:
        required: true
        default: staging
        type: string

    secrets:
      docker_token:
        required: true

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Build Info
        run: |
          echo "Building ${{ inputs.app_name }} for ${{ inputs.environment }}"
          echo "Docker token is set: ${{ secrets.docker_token != '' }}"
```

# Task 3: Caller Workflow

## File

```text
.github/workflows/call-build.yml
```

## YAML

```yaml
name: Call Reusable Workflow

on:
  push:
    branches:
      - main

jobs:
  build:
    uses: ./.github/workflows/reusable-build.yml

    with:
      app_name: "my-web-app"
      environment: "production"

    secrets:
      docker_token: ${{ secrets.DOCKER_TOKEN }}
```

## Result

1. Push to main triggers the workflow.
2. Caller workflow invokes reusable-build.yml.
3. Inputs are passed successfully.
4. Secret is consumed securely.

# Task 4: Reusable Workflow Outputs

## Updated Reusable Workflow

```yaml
name: Reusable Build

on:
  workflow_call:
    inputs:
      app_name:
        required: true
        type: string

      environment:
        required: true
        type: string

    secrets:
      docker_token:
        required: true

    outputs:
      build_version:
        description: Build version
        value: ${{ jobs.build.outputs.build_version }}

jobs:
  build:
    runs-on: ubuntu-latest

    outputs:
      build_version: ${{ steps.version.outputs.build_version }}

    steps:
      - uses: actions/checkout@v4

      - name: Generate Version
        id: version
        run: |
          echo "build_version=v1.0-${GITHUB_SHA::7}" >> $GITHUB_OUTPUT
```

## Updated Caller Workflow

```yaml
name: Call Reusable Workflow

on:
  push:
    branches:
      - main

jobs:
  build:
    uses: ./.github/workflows/reusable-build.yml

    with:
      app_name: "my-web-app"
      environment: "production"

    secrets:
      docker_token: ${{ secrets.DOCKER_TOKEN }}

  show-version:
    runs-on: ubuntu-latest
    needs: build

    steps:
      - name: Print Version
        run: |
          echo "Version: ${{ needs.build.outputs.build_version }}"
```

## Result

The second job successfully reads and prints the version generated by the reusable workflow.

# Task 5: Create a Composite Action

## File Structure

```text
.github/
└── actions/
    └── setup-and-greet/
        └── action.yml
```

## action.yml

```yaml
name: Setup And Greet

inputs:
  name:
    required: true

  language:
    required: false
    default: en

outputs:
  greeted:
    value: ${{ steps.greet.outputs.greeted }}

runs:
  using: composite

  steps:
    - name: Greeting
      id: greet
      shell: bash
      run: |
        if [ "${{ inputs.language }}" = "en" ]; then
          echo "Hello ${{ inputs.name }}"
        else
          echo "Namaste ${{ inputs.name }}"
        fi

        echo "Date: $(date)"
        echo "OS: $RUNNER_OS"

        echo "greeted=true" >> $GITHUB_OUTPUT
```

## Workflow Using Composite Action

```yaml
name: Composite Action Demo

on:
  workflow_dispatch:

jobs:
  greet:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Run Custom Action
        uses: ./.github/actions/setup-and-greet

        with:
          name: Vinit
          language: en

      - name: Completed
        run: echo "Composite Action Executed"
```

# Task 6: Reusable Workflow vs Composite Action

| Feature | Reusable Workflow | Composite Action |
|----------|----------|----------|
| Triggered By | workflow_call | uses: |
| Can Contain Jobs | Yes | No |
| Can Contain Multiple Steps | Yes | Yes |
| Lives In | .github/workflows | Any folder with action.yml |
| Accept Secrets Directly | Yes | No |
| Best For | Entire CI/CD processes | Reusable step groups |

---

# Key Learnings

1. Reusable workflows eliminate duplicated CI/CD logic across repositories.
2. Outputs allow workflows to pass data between jobs and workflows.
3. Composite actions are useful for packaging reusable step sequences into a single action.
